# 长期记忆生命周期与权限审计实验结果

- 日期：2026-07-11
- 命令：`uv run python docs/experiments/memory-lifecycle-audit/memory_lifecycle_audit.py`
- 运行状态：通过
- 依赖：Python 标准库；无模型调用；无真实 memory framework

## 汇总结果

| 策略 | passed operations | failed operations | trace 泄露合成敏感值 | 说明 |
| --- | ---: | --- | --- | --- |
| `naive_memory` | 1 / 6 | `inspect_other_user`、`edit_preference`、`delete_secret`、`recall_after_delete`、`cross_user_recall` | true | 只通过了查看自己记忆；跨用户隔离、编辑历史、删除、删除后召回和 trace 脱敏都失败。 |
| `governed_memory` | 6 / 6 | 无 | false | 阻断跨用户访问，编辑时 invalidates 旧版本，删除敏感记录后不再召回，并对 trace 脱敏。 |

## 操作结果

| operation | `naive_memory` | `governed_memory` | 观察 |
| --- | --- | --- | --- |
| `inspect_own` | pass | pass | 两种策略都允许用户查看自己的 active memory。 |
| `inspect_other_user` | fail | pass | naive 策略把其他用户的记忆也暴露给当前用户；governed 策略返回 `blocked_cross_user`。 |
| `edit_preference` | fail | pass | naive 策略直接覆盖偏好且没有历史；governed 策略把旧偏好标为 `invalidated`，新偏好版本递增。 |
| `delete_secret` | fail | pass | naive 策略不支持删除；governed 策略把敏感记录标为 `deleted` 并从 active memory 移除。 |
| `recall_after_delete` | fail | pass | naive 策略删除后仍召回敏感记录；governed 策略返回 `secret_absent`。 |
| `cross_user_recall` | fail | pass | naive 策略可跨用户召回；governed 策略阻断跨用户 recall。 |

## 关键观察

- 只做“写入守门”不够。长期记忆还需要 inspect、edit、delete、recall 的完整生命周期权限检查。
- 删除不是 UI 上隐藏一条记录。删除后的记录必须退出 active memory 和后续 recall；历史记录也需要脱敏。
- 跨用户隔离应该在 memory store / retrieval 层实现，而不是只依赖模型不要引用其他用户信息。
- trace 是治理的一部分。即使最终回答不泄露敏感值，trace / audit log 也可能成为泄露面。
- 编辑偏好或事实时，保留 invalidated history 比静默覆盖更适合审计和纠错。

## 支撑的结论

- 长期记忆系统应提供用户可查看、可编辑、可删除的能力。
- Memory recall 应执行 owner / target user 权限检查，避免跨用户污染和泄露。
- 删除和编辑应影响后续 recall，而不是只修改显示层。
- Trace、history 和 debug log 应默认脱敏敏感字段。

## 不能支撑的结论

- 不能证明长期记忆提升任务表现。
- 不能证明任何真实 memory framework 默认满足这些生命周期要求。
- 不能覆盖真实鉴权、组织级权限、用户 UI、长期保留策略、隐私合规、成本、延迟或真实模型误用记忆概率。
