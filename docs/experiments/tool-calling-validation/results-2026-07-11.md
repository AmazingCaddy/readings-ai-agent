# Tool Calling 参数校验与重试实验结果

- 日期：2026-07-11
- 命令：`uv run python docs/experiments/tool-calling-validation/tool_calling_validation.py`
- 运行状态：通过
- 依赖：Python 标准库；无模型调用；无网络请求；无真实 Function Calling API

## 观察结果

脚本输出状态为 `ok`，最终响应为 `Tokyo is 21 C and cloudy.`。

Trace 包含 5 个事件：

| step | actor | event | 说明 |
| --- | --- | --- | --- |
| 1 | model | `tool_call_requested` | fake model 请求 `get_weather`，但参数为 `unit=kelvin`。 |
| 1 | application | `tool_validation_failed` | 应用层拒绝执行，返回 `unit must be one of: celsius, fahrenheit`。 |
| 2 | model | `tool_call_requested` | fake model 在错误回传后改为 `unit=celsius`。 |
| 2 | application | `tool_executed` | 应用层执行通过校验的工具调用。 |
| 3 | model | `final_response` | fake model 基于工具结果生成最终响应。 |

## 支撑的结论

- Function Calling / Tool Calling 的执行边界在应用侧：模型请求工具调用，应用程序决定是否执行。
- 参数 schema 之外仍可能需要业务规则校验，例如 `unit` 只能属于允许集合。
- 工具错误应包含可操作信息，才能进入后续修正循环。
- 重试循环需要最大轮数；本实验设置 `max_turns=4`，避免无限重试。
- trace 至少应记录 tool call、validation failure、tool execution 和 final response。

## 不能支撑的结论

- 不能证明真实模型稳定修正参数。
- 不能证明 OpenAI、LangGraph、Agents SDK 或其他框架的具体错误格式。
- 不能覆盖权限确认、有副作用工具、工具结果 prompt injection、并发工具调用或真实 API latency/cost。
