# 多 Agent 与 Flow 控制对比实验结果

- 日期：2026-07-11
- 命令：`uv run python docs/experiments/multi-agent-comparison/multi_agent_comparison.py`
- 运行状态：通过
- 依赖：Python 标准库；无模型调用；无真实 Agent framework

## 汇总结果

| 策略 | 成功数 | 工具调用数 | messages | conflicts | duplicate reads | 失败任务 | 说明 |
| --- | ---: | ---: | ---: | ---: | ---: | --- | --- |
| `single_checklist` | 3/3 | 6 | 0 | 0 | 0 | 无 | 任务边界清楚时，单流程足够且开销最低。 |
| `ungoverned_multi_agent` | 1/3 | 7 | 7 | 1 | 3 | `TASK-2`、`TASK-3` | 角色自行读取导致重复、缺 `cost.md` / `feedback.md`，并产生未解决冲突。 |
| `flow_controlled_multi_agent` | 3/3 | 6 | 9 | 0 | 0 | 无 | Flow 明确 required docs 和角色边界后成功，但仍有消息协调开销。 |

## 关键观察

- `single_checklist` 对三类任务都成功，说明清晰任务不一定需要多 Agent。
- `ungoverned_multi_agent` 多次重复读取 `spec.md`，产生 3 次 duplicate reads。
- `ungoverned_multi_agent` 在外部发布任务中读取了 `security.md`，但漏掉 `cost.md`，并留下 `conflict unresolved`。
- `ungoverned_multi_agent` 在初学者教程任务中漏掉 `feedback.md`，无法写出 beginners/examples 要点。
- `flow_controlled_multi_agent` 通过 `plan_created` 明确 required docs，通过 `review_completed` 检查 missing docs / missing terms，成功率恢复到 3/3。
- `flow_controlled_multi_agent` 即使成功，也产生 9 条 messages，说明角色协作有通信成本。

## 支撑的结论

- 多 Agent 是一种编排选择，不是复杂任务的默认升级路径。
- 如果任务能用单流程 checklist 稳定完成，多 Agent 可能只增加通信和调试成本。
- 多 Agent 更需要 Flow / workflow 控制 required evidence、角色边界、冲突处理和 review trace。

## 不能支撑的结论

- 不能证明真实 AutoGen、CrewAI、LangGraph 或其他框架的表现。
- 不能证明多 Agent 在真实研究、写作或工程任务中一定更差或更好。
- 不能覆盖真实模型 token、latency、成本、工具异常、长期状态、人工评审或复杂冲突合并。
