## 支撑的结论

- Repo Issue Agent 进阶练习可以先用本地 toy repo 和固定 workflow 建立对照组。
- 一个可用 toy repo 至少应记录 issue、失败测试、读取文件、diff、复跑测试、是否修改测试、trajectory 和敏感信息脱敏。
- workflow-agent hybrid 的教学重点可以先放在“模型/建议者只提下一步，命令由应用或人工审批执行”这个控制边界上；本次确定性 baseline 可记录建议、批准命令、拒绝命令、scoped diff、测试结果和 secret 脱敏。
- 固定 workflow baseline 能帮助判断后续 coding agent 是否真的带来收益，而不是只看“最后测试通过”。

## 不能支撑的结论

- 本次没有运行 mini-SWE-agent、SWE-agent、OpenAI Agents SDK、LangGraph coding agent 或真实模型；workflow-agent hybrid 是确定性 suggestion stub，不是模型驱动 agent。
- 本次不能证明任何 coding agent 能修复 repo issue，也不能证明 bash-only agent、confirm 模式、sandbox 或某个模型默认可靠。
- 本次没有记录 token、模型调用成本、真实 agent latency、人工确认负担、容器 sandbox 隔离效果或 trajectory browser 行为。

## 后续

下一步应在同一个 toy repo 上增加 mini-SWE-agent confirm-mode 和可选 SWE-agent run；如果引入真实模型驱动的 workflow-agent hybrid，也要记录模型建议、命令确认、失败/重试、diff、测试输出、token/latency/cost 和是否尝试越权读取或修改测试。
