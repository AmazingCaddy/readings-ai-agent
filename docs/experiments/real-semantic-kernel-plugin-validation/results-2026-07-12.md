## Case 结果

| case | 结果 | 关键观察 |
| --- | --- | --- |
| `metadata_exposes_plugin_functions` | pass | `@kernel_function` 暴露 `refund.read_order` / `refund.issue_refund` metadata，`amount` 参数为 required float。 |
| `read_function_invokes` | pass | `Kernel.invoke()` 可调用只读 native function。 |

| `invalid_argument_type_rejected` | pass | `amount="not-a-number"` 时，Semantic Kernel runtime 拒绝调用。 |
| `coercible_argument_invokes` | pass | `amount="12.5"` 可被解析为 float 并执行。 |

## 支撑的结论

- Semantic Kernel native plugin 可以把 Python method 注册为 kernel function，并暴露 plugin/function/parameter metadata。
- Semantic Kernel runtime 会处理缺少 required 参数和不可解析类型的调用错误。
- 可解析的字符串数值可能被转换后执行，因此业务校验不能只看原始 JSON 类型。
- 高风险写函数的审批可以在应用层 wrapper 中完成，并在未审批时不转发给 kernel。
- 本地 trace 可以记录 policy reject、kernel invoke 和 kernel error，同时避免记录示例 secret。

## 不能支撑的结论

- 不能证明真实模型会正确选择 function、修正参数或遵守审批要求。
- 不能证明 Semantic Kernel OpenAPI plugin、MCP plugin、Agent Framework、Process Framework 或 HITL UI 的真实行为。
- 不能证明框架默认提供高风险工具审批、参数快照、幂等恢复、trace 脱敏、成本控制、低延迟或生产安全。
