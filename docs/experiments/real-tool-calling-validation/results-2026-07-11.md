## 本地 control fixtures

无 API key 时，脚本运行一个 deterministic validation/retry fixture：

- 第一轮 fake response 返回 `get_weather(city=Tokyo, unit=kelvin)`；这个参数满足工具 schema 中的 string 类型，但违反应用层业务规则。
- 应用层 validator 返回 `unit must be one of: celsius, fahrenheit`，不执行工具。
- 第二轮 fake response 返回 `get_weather(city=Tokyo, unit=celsius)`；应用层校验通过后才执行 toy weather tool。

本次结果中 `tool_calls_seen=2`、`validation_failed_count=1`、`corrected_tool_call_count=1`、`tool_execution_count=1`，全部符合预期。

## 已准备的真实验证入口

脚本已准备一个最小 Responses API tool-calling 流程：

- 请求模型回答天气问题，并要求使用 `get_weather` function tool。
- 工具 schema 只要求 `unit` 是 string。
- 应用层业务校验只允许 `celsius` / `fahrenheit`，用于测试 schema-valid 但业务非法的 `kelvin`。
- 把 validation error 作为 `function_call_output` 回传给模型。
- 观察模型是否修正参数、继续发起非法 tool call，或给出最终说明。

每个 completed run 会记录：

- request created。
- first response id。
- tool call name、call id 和 arguments。
- application validation error 或 tool execution result。
- second response id。
- second tool call 或 final response。
- status：`no_tool_call`、`corrected_tool_call`、`repeated_invalid_tool_call` 或 `final_response_after_tool_feedback`。

## 当前支撑的结论

- 可支撑：真实 Responses API tool-calling 参数校验入口已准备；无 API key 时运行本地 deterministic control，验证 tool-call parser、应用层业务校验、错误反馈形状和 toy tool execution 边界。
- 不能支撑：当前没有真实 API completed run；不能证明模型会稳定生成 tool call、遵循 schema、根据 validation error 修正参数、正确处理 tool output、控制成本/延迟，或证明任何框架默认错误处理可靠。
