# RAG 最小 Pipeline 与 Citation 实验结果

- 日期：2026-07-11
- 命令：`uv run python docs/experiments/rag-pipeline/rag_pipeline_simulation.py`
- 运行状态：通过
- 依赖：Python 标准库；无模型调用；无网络请求；无 embedding / vector store
- 控制：`deterministic_rag_pipeline_and_retrieval_strategy_fixtures`

## 观察结果

脚本处理 3 个文档，默认 chunk size 下生成 3 个 chunk。Trace 覆盖 `chunk`、`retrieve`、`synthesize` 三类阶段。扩展后的 `strategy_audit` 覆盖 3 个 retrieval case、4 个 retrieval strategy、12 个结果，其中 4 个 expected failure 都按预期暴露，`expectations_met=true`。

### Case 1: provenance / source tracing

- 问题：`How does RAG help with provenance and source tracing?`
- 检索结果：`llamaindex-rag#1` 和 `rag-paper#1`。
- 输出：`RAG helps with provenance by retrieving external source chunks and attaching citations to the answer.`
- citations：包含 `chunk_id`、`source_id`、`title`、`url`。
- grounded：`true`

### Case 2: RAG vs long-term memory

- 问题：`How is RAG different from long-term memory?`
- 检索结果：`langgraph-memory#1` 和 `llamaindex-rag#1`。
- 输出：`RAG retrieves external knowledge, while long-term memory stores cross-session user or application data that needs governance.`
- citations：包含 memory 和 RAG 两类来源。
- grounded：`true`

### Case 3: unsupported question

- 问题：`What is the project deployment command?`
- 检索结果：空。
- 输出：`I do not have enough retrieved evidence to answer this question.`
- citations：空。
- grounded：`false`

## Retrieval Strategy Audit

| strategy | 目的 | 观察结果 |
| --- | --- | --- |
| `balanced_top2` | 默认关键词 baseline，top-k=2。 | `multi_source_provenance`、`memory_governance` 和 unsupported case 都符合预期。 |
| `top1_misses_multi_source` | 暴露 top-k 过小的多来源召回风险。 | 在 provenance 和 memory 两个需要多来源的 case 中只召回 1/2 个 expected source；unsupported case 仍正确空召回。 |
| `wrong_metadata_filter_hides_memory` | 暴露 metadata filter 错配会隐藏必要来源。 | 在 memory case 中过滤掉 `langgraph-memory`，recall 只有 0.5；其他不依赖该来源的 case 不应失败。 |
| `fine_chunks_top3_rerank_still_needs_eval` | 暴露细 chunk 和 rerank terms 仍需 eval。 | provenance case 通过；memory case 漏掉 `llamaindex-rag`，说明细 chunk / rerank 不是自动更好。 |

## Strategy Audit 关键字段

- `expected_source_ids`：该问题最少需要召回的来源集合。
- `retrieved_source_ids`：实际被选中的来源集合。
- `recall_expected_sources`：本地控制用的来源召回比例。
- `expected_outcome`：本 fixture 预期通过或预期失败。
- `expectation_met`：策略是否暴露了预期行为。

## 支撑的结论

- 最小 RAG trace 应至少记录 chunk、retrieve 和 synthesize 阶段。
- Chunk 应保留 `chunk_id`、`source_id`、`title`、`url` 等可追溯 metadata。
- Answer citations 应绑定到具体 chunk，而不是只给一个泛泛文档链接。
- 没有检索证据的问题应保守拒答或标记为 `grounded=false`，不能编造答案。
- Chunk size、top-k、metadata filter 和 rerank/filter 设置都应进入 retrieval eval 记录；top-k 过小、filter 错配和细 chunk + rerank 都可能漏掉必要来源。

## 不能支撑的结论

- 不能证明关键词检索优于 embedding、hybrid retrieval 或 rerank。
- 不能证明任何 chunk size、top-k 或检索策略在真实任务中最优。
- 不能证明 LLM 合成答案时会忠实引用来源。
- 不能证明真实 RAG 系统的 latency、token cost 或 citation correctness。
- 不能证明 OpenAI File Search、LlamaIndex、vector DB 或真实 reranker 的默认策略表现。
