# Planner / Executor 与单循环对比实验结果

- 日期：2026-07-11
- 命令：`uv run python docs/experiments/planner-executor-comparison/planner_executor_comparison.py`
- 运行状态：通过
- 依赖：Python 标准库；无模型调用；无真实 Agent framework

## 汇总结果

| 策略 | 成功数 | 工具调用数 | plan steps | replans | 失败任务 | 说明 |
| --- | ---: | ---: | ---: | ---: | --- | --- |
| `single_checklist_loop` | 3/3 | 9 | 0 | 0 | 无 | 固定 checklist 覆盖了三类任务，成本可预测。 |
| `planner_executor_once` | 2/3 | 8 | 8 | 0 | `TASK-3` | 一次性计划漏掉 migration 文件，输出缺少关键证据。 |
| `planner_executor_with_feedback` | 3/3 | 9 | 9 | 1 | 无 | executor 发现 missing evidence 后重规划，补读 migration 文件。 |

## 关键观察

- `single_checklist_loop` 在任务边界清楚时足够完成，不需要额外 planner。
- `planner_executor_once` 在 `TASK-3` 中只读取 issue、service code 和 schema，漏掉 `migrations/2026_07_add_customer_id.sql`。
- `planner_executor_once` 的输出只能说 “migration evidence is missing”，不能完整支撑 root cause。
- `planner_executor_with_feedback` 记录 `validation_failed`，发现 missing evidence 后记录 `plan_revised` 并补读 migration 文件。
- 带反馈的 planner/executor 成功率恢复到 3/3，但 plan steps 和 trace 复杂度增加。

## 支撑的结论

- Planner / Executor 的价值不在于“先写计划”本身，而在于计划可执行、执行可校验、失败可反馈。
- 一次性计划可能遗漏关键证据，导致错误或弱结论传播。
- 对比规划架构时，需要记录 plan steps、replans、missing evidence、工具调用数和失败原因。

## 不能支撑的结论

- 不能证明 Planner / Executor 在真实任务中优于单循环或 workflow。
- 不能证明任何真实模型可以稳定生成可执行计划或可靠重规划。
- 不能覆盖 token、latency、成本、权限确认、工具异常、人类评审或多 Agent 协作。

## 后续实验

- 用真实模型和真实 repo issue 比较 single loop、planner/executor、planner/executor with feedback 和 reflection retry。
- 增加工具失败、权限拒绝、冲突证据和人工确认场景。
