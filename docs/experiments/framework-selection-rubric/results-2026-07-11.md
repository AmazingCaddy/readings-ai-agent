## 关键观察

- Rubric 能把“任务需要什么”拆成 `required`、`nice_to_have` 和 `avoid`，比直接问“哪个框架最好”更可检查。
- `T4-role-review` 出现 AutoGen / CrewAI tie，说明多 Agent 框架选择不能只看能力标签，还要继续比较真实实现、成本、trace、冲突处理和维护负担。
- `T5-enterprise-plugin` 虽然把 Semantic Kernel 排第一，但结果同时保留 process experimental / concept load caution，避免把企业集成能力写成默认优势。
- 低分框架不代表“不好”，只表示它和当前任务画像不匹配。

## 支撑的结论

- 框架比较应从任务难点出发，至少记录 required capabilities、nice-to-have、avoid、missing required 和 cautions。
- 框架文档可以支持能力定位，但不能单独推出真实项目中的最佳选择。
- Multi-agent、enterprise integration 和 graph/workflow 都会引入额外学习成本或调试成本，应在选择时显式记录。

## 不能支撑的结论

- 不能证明任何框架真实更快、更便宜或更可靠。
- 不能证明真实 tracing、permission、HITL、RAG citation、tool error recovery 或 deployment 体验。
- 不能替代同一任务在真实 OpenAI Agents SDK、LangGraph、LlamaIndex、AutoGen、CrewAI 和 Semantic Kernel 下的横向实验。

## 后续实验

- 选择一个固定任务：检索资料、调用只读工具、生成带来源答案、必要时请求人工确认。
- 用 2-3 个框架先做真实最小实现，记录依赖、实现时间、trace 可读性、权限/人工确认、错误恢复、token/latency/cost。
- 再扩展到完整框架横向比较，避免一次性引入太多依赖和 API 成本。
