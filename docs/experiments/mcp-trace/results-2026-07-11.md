# MCP 最小 Trace 实验结果

- 日期：2026-07-11
- 命令：`uv run python docs/experiments/mcp-trace/mcp_trace_simulation.py`
- 运行状态：通过
- 依赖：Python 标准库；无模型调用；无网络请求；无真实 MCP SDK

## 观察结果

脚本输出了 8 个 trace event：

| 方法 | actor | decision | 说明 |
| --- | --- | --- | --- |
| `roots/list` | client | `returned_allowed_roots` | client 暴露 1 个 workspace root。 |
| `tools/list` | server | `returned_tool_catalog` | server 暴露 2 个 tools。 |
| `tools/call.approval` | host | `approved` | host 允许只读工具 `lookup_policy_section`。 |
| `tools/call` | server | `returned_result` | server 返回只读工具结果。 |
| `tools/call.approval` | host | `rejected` | host 拒绝模拟写工具 `write_release_note`。 |
| `resources/list` | server | `returned_resource_catalog` | server 暴露 2 个 resources。 |
| `resources/read.review` | host | `ordinary_context` | host 将普通资源标为 ordinary context。 |
| `resources/read.review` | host | `untrusted_prompt_injection_candidate` | host 将包含“ignore previous instructions”和“secret”的资源标为 prompt injection 候选。 |

## 支撑的结论

- 最小 trace 至少应记录 actor、method、decision、关键参数摘要和时间戳。
- `tools/list` / `tools/call` 和 `resources/list` / `resources/read` 应分开记录，因为工具调用和资源读取的风险不同。
- 高风险 tool call 的批准或拒绝应记录在 host 层，而不是假设 server 或模型自动负责。
- 资源内容即使只是“读取”，也可能包含 prompt injection 风格文本，因此 host 需要把 resource review 与 tool approval 分开设计。
- Roots 可以记录可见范围，但本实验不提供真正 sandbox，因此不能把 roots 写成安全隔离机制。

## 不能支撑的结论

- 不能证明真实 host 默认会记录这些字段；官方 MCP Python SDK / FastMCP stdio 对照已另见 Real MCP SDK Trace 结果。
- 不能证明 URL mode elicitation、OAuth authorization、token audience validation 或 token redaction 的真实行为。
- 不能证明模型会稳定遵守 host 对恶意资源的标记。

## 后续实验

- 用真实 host UI 接一个只读 server，记录工具展示、用户拒绝、resource review 和审计 trace。
- 加入 URL mode elicitation 或模拟 authorization flow，验证用户同意、URL 展示和敏感日志脱敏。
- 在至少两个 host 中比较 tool confirmation、roots 展示、resource review 和 audit log 差异。
