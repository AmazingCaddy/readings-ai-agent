## 支撑的结论

- 成本、延迟和限流不是上线后才补的附属指标；最小记录应包含 model、token、request count、rate-limit headers、retry、latency distribution、budget gate 和降级动作。
- 重试策略要有上限、backoff、jitter 和 stop reason；失败请求也可能影响限流预算，不能把无限重试当作可靠性方案。
- 降低成本或延迟的策略必须同时记录质量检查，否则“更便宜 / 更快”可能只是牺牲输出质量或完整性。
- Batch、Flex 和 Prompt Caching 需要各自的观测字段：Batch status / `custom_id` / expiration，Flex resource unavailable / fallback，Prompt Caching `cached_tokens` / `cache_write_tokens` / miss reason。
- Budget threshold 应连接 stop、degrade 或 escalation，而不是只记录 spent 数字。

## 不能支撑的结论

- 不能证明真实 API 成本、P95 latency、吞吐、缓存命中率、Batch 完成率、Flex 可用性或优化收益。
- 不能证明较小模型、减少 output tokens、Batch、Flex、Prompt Caching、streaming 或并行化在任何具体任务中默认更好。
- 不能替代真实 quality eval、真实 rate-limit headers、真实 pricing、真实 failure samples 或用户体验测试。
