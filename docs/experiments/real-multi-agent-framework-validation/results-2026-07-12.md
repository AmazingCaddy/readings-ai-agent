## 框架观察

| 框架 | 版本 | 本地 runtime surface | 本次观察 |
| --- | --- | --- | --- |
| AutoGen AgentChat | 0.7.5 | `AssistantAgent` + `RoundRobinGroupChat` + `TextMentionTermination` | 产生 3 条 team transcript message：user、researcher、reviewer；reviewer 输出 `feedback.md missing` 并触发 `TERMINATE`；发布 trace 未泄露示例 secret |
| CrewAI | 1.15.2 | `Agent` + `Task` + `Crew(process=sequential)` | 产生 2 个 task output：Researcher 和 Reviewer；Reviewer 输出 `feedback.md missing`；发布 trace 未泄露示例 secret |

## 能力边界

框架直接提供的能力：

- AutoGen AgentChat：agent abstraction、round-robin team scheduling、team termination condition、message transcript。
- CrewAI：agent abstraction、task abstraction、sequential crew process、task output chaining through context。

应用层仍负责的能力：

- fake model / fake LLM response 设计。
- evidence policy 和 missing-evidence rubric。
- 是否允许进入最终正文或审批下一步。
- trace 发布前脱敏和敏感数据审计。

## 结论

本次结果可以支撑一个窄结论：AutoGen AgentChat 和 CrewAI 都能在本地 runtime 中表达 researcher/reviewer 式多 Agent / 多角色 workflow，并产生可检查的 transcript 或 task output。因此正文可以把多 Agent 框架写成“可用的编排表面”，而不是只停留在文档声明。

本次结果不能支撑更宽结论：它不调用真实模型，不比较质量，不记录 token/cost/latency，不验证生产 tracing、真实权限 UI、部署恢复、人工评审负担或真实冲突合并能力。多 Agent 是否值得引入，仍必须用同一任务、同一验收标准和真实模型/成本/trace 做对照。
