# Workflow、Hybrid 与 ReAct-like Tool Loop 对比实验结果

- 日期：2026-07-11
- 命令：`uv run python docs/experiments/workflow-agent-comparison/workflow_agent_comparison.py`
- 运行状态：通过
- 依赖：Python 标准库；无模型调用；无真实 Agent framework

## 汇总结果

| 策略 | 成功数 | 工具调用数 | 失败 issue | 说明 |
| --- | ---: | ---: | --- | --- |
| `fixed_workflow` | 2/3 | 3 | `ISSUE-2` | 简单文档和 feature triage 足够，但没有读取 log/config/deploy evidence，无法定位登录超时根因。 |
| `workflow_agent_hybrid` | 3/3 | 7 | 无 | 固定分类后只在 bug 分支读取证据，定位 cache config port mismatch。 |
| `react_like_loop` | 3/3 | 9 | 无 | 能动态读取证据；但在 feature 任务中也额外搜索和读取产品文档，工具调用更多。 |

## 关键观察

- `fixed_workflow` 对步骤固定、只需 issue 内容的任务成本最低。
- `fixed_workflow` 在 `ISSUE-2` 中只能输出“login timeout bug”，不能给出 `6379` / `6380` 的配置证据。
- `workflow_agent_hybrid` 通过固定路由限制动态查询范围：只有 bug 分支读取 `logs/auth.log`、`services/auth/config.py` 和 `deploy/cache.env`。
- `react_like_loop` 同样能完成 bug root cause 分析，但总工具调用数更高。
- `react_like_loop` 对 feature request 额外读取 `product/export.md`，说明动态循环可能带来不必要查询；它需要预算、停止条件和 trace 审查。

## 支撑的结论

- Agent 和 Workflow 应理解为控制权和编排方式的连续谱，而不是互斥选择。
- 初学者可以先做固定 workflow；当任务确实需要中间证据和动态查询，再引入 workflow-agent hybrid 或 Agent loop。
- 比较架构时应同时记录成功率、工具调用次数、失败原因、trace 和停止条件。

## 不能支撑的结论

- 不能证明 ReAct 在真实任务中稳定优于固定 workflow。
- 不能证明 workflow-agent hybrid 在所有任务中更好。
- 不能证明任何真实模型、真实框架或真实代码库上的成本、延迟、权限和错误恢复表现。
