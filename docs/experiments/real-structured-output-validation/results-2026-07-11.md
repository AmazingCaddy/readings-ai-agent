## 本地 control fixtures

无 API key 时，脚本运行 3 个 deterministic fixtures：

- `free_text`：自然语言文本无法作为 JSON 解析，结果为 `parse_failed`。
- `json_mode`：合法 JSON 且 schema-valid，但自动批准 90 天旧订单且 `needs_human=false`，semantic validation 判为 `invalid`。
- `json_schema`：合法 JSON、schema-valid，并正确升级到人工 review，结果为 `ok`。

本次结果中 `schema_valid_count=2`、`semantic_valid_count=1`、`parse_failed_count=1`，全部符合预期。

## 已准备的真实验证入口

脚本已准备 3 种输出模式对照：

- `free_text`：不设置结构化输出格式，观察解析失败或非 JSON 输出。
- `json_mode`：使用 JSON object 格式，观察合法 JSON 与 schema / semantic validation 的差异。
- `json_schema`：使用 strict JSON Schema，观察 schema adherence、refusal 和业务语义校验。

每个 completed case 会记录：

- `schema_valid`。
- `semantic_valid`。
- parsed payload。
- schema errors。
- semantic errors。
- response id。

## 当前支撑的结论

- 可支撑：真实 Responses API Structured Outputs / JSON mode 对照入口已准备；无 API key 时运行本地 deterministic control，验证 parser、schema validator 和 semantic validator 能区分 parse failure、schema-valid 但业务错误、schema-valid 且业务正确三类结果。
- 不能支撑：当前没有真实 API completed run；不能证明真实 free text、JSON mode、Structured Outputs、refusal、retry loop、跨模型稳定性、成本或延迟表现。
