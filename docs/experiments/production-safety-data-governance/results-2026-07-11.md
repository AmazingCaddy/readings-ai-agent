## 支撑的结论

- 生产安全和数据治理检查表不能只写“接入 moderation”或“API 数据默认不训练”。至少要区分 detection signal、policy action、human review、失败降级、应用日志、数据保留、第三方工具数据流和删除路径。
- Tool-calling moderation 覆盖范围需要拆开记录：tool arguments / outputs 可以进入 moderation workflow，但 tool names、descriptions、schemas 和 response-format schemas 仍需要人工或应用层审查。
- Streaming 场景需要单独记录 partial output policy，因为最终 moderation scores 不是 partial deltas 的实时保护层。
- `safety_identifier`、API key revoke、abuse logs、application state、remote MCP 数据流、hosted tool state 和 data residency 都应进入上线前检查表。
- Remote MCP、hosted execution、file/vector store、prompt caching 和 browser/computer-use 不能只写成“工具已启用”；它们各自有对象、第三方、保留、删除、trace 和审批字段。
- Red-team 和用户举报不是一次性流程，确认的问题应回写到 regression set。

## 不能支撑的结论

- 不能证明真实 Moderation API、检测层、HITL、guardrail、abuse monitoring、ZDR/MAM、data residency、API key revoke 或第三方工具数据治理有效。
- 不能证明任何真实 project 的配置已经满足合规要求。
- 不能记录真实误报、漏报、成本、延迟、人工复核负担、第三方保留行为、对象删除一致性、trace 脱敏有效性或跨地区能力差异。
