## 关键观察

- Human approval 不是一个布尔字段。系统需要保存审批对象、审批人、审批状态、参数快照、执行状态和审计事件。
- Pause/resume 需要幂等设计。用户刷新页面、重试请求或 worker 重放消息时，同一个 approval id 不应重复产生副作用。
- 审批的是一组具体参数。恢复执行前如果参数变化，应要求重新审批或直接阻断。
- 拒绝后恢复执行必须被阻断。否则“拒绝”只是在 UI 上显示，不能构成安全边界。
- Trace 需要记录恢复和阻断原因，但不能把敏感字段原样写入审计日志。

## 支撑的结论

- 高风险工具的人工审批应设计成可审计、可恢复、可幂等的执行流程。
- 审批状态恢复至少要校验审批状态、执行状态和参数快照。
- `require_approval`、`approved`、`rejected`、`executed` 和 `already_executed` 应是不同状态，而不是混成“用户确认过”。
- Trace 应记录 approval id、decision、resume result 和阻断原因，同时对敏感字段脱敏。

## 不能支撑的结论

- 不能证明任何真实框架的 HITL pause/resume 行为。
- 不能覆盖真实工具副作用、真实用户界面、组织权限、审批超时、异步队列、并发 race、成本、延迟或人工审批负担。
- 不能证明真实模型会稳定生成可审批的工具参数。
