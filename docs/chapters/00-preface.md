# 序言：如何学习 AI Agent

## 本章适合谁

如果你刚开始系统学习 AI Agent，或者已经看过一些文章但分不清 Agent、Workflow、Tool Use、RAG、Memory、MCP、Eval 的边界，这一章适合先读。

这本手册默认你了解基本编程概念，但不要求你已经读过 Agent 论文或用过任何 Agent 框架。

## 你会学到什么

- 这本手册解决什么问题。
- 为什么不能只靠网上文章学习 Agent。
- 如何区分“资料已复核”和“结论已验证”。
- 应该按什么顺序学习。
- 后续章节如何使用 references。

## 先用一句话理解

学习 AI Agent 的关键不是收集最多资料，而是建立一套能判断资料是否可靠、概念是否清楚、工程建议是否有边界的学习系统。

## 为什么需要这本手册

AI Agent 相关资料非常多，但初学者最容易遇到三个问题。

第一，很多资料把概念混在一起。比如把 Agent、Workflow、RAG 应用、聊天机器人都叫 Agent。这样会让你误以为所有 LLM 应用都应该做成高自治 Agent。

第二，很多文章只展示 demo，不解释失败模式。Agent demo 可以看起来很聪明，但真实系统还要处理工具错误、权限边界、上下文污染、成本、延迟和评测。

第三，资料更新很快。OpenAI、MCP、LangGraph、LlamaIndex、AutoGen 等文档持续变化，旧文章里的 API、术语和最佳实践可能已经过时。

因此，这本手册采用“章节正文 + source cards + 证据台账”的方式组织内容：正文负责讲清楚，source cards 负责记录来源，证据台账负责控制哪些结论可以写成确定性说法。

## 学习顺序

建议按下面顺序学习，而不是一开始就挑框架。

1. 先读 [术语边界表](../glossary.md)，建立概念边界。
2. 再读 [初学者学习路径](../learning-path.md)，了解整体路线。
3. 按章节顺序学习：先理解 Agent 是什么，再学 LLM、工具调用、架构、MCP、RAG、Memory、Eval 和生产化。
4. 每章读完后，看 References 和待验证问题。
5. 如果你要做项目，再回到 source cards 和实验清单。

## 如何判断资料可靠性

这本手册使用四级可信度。

- A 级：官方文档、论文、协议规范、源码、可复现实验。
- B 级：知名团队工程实践，有代码、数据或清晰上下文。
- C 级：个人博客、课程、访谈、总结文章，需要交叉验证。
- D 级：营销文、无来源观点、过时内容，只能作为线索。

正文默认只采用 A/B 级资料。C 级资料只能作为补充观点，D 级资料不进入正文。

但要注意：来源可信不等于结论已经验证。比如一篇论文是 A 级来源，但论文中的实验结论仍然只适用于它的任务、模型、数据和评估设置。把论文结论直接泛化到所有 Agent 系统是不严谨的。

## 本手册如何使用 References

References 不是装饰。每章的关键定义、重要结论、工程建议和争议点都应该能追溯到资料来源。

本手册会区分三种状态：

- 链接已复核：确认来源页面可访问，但尚未精读。
- 元数据已复核：确认论文标题、作者、发布时间、摘要等基础信息。
- 可入正文：结论已有清晰支撑，并有边界说明或交叉验证。

如果某个结论仍然在候选状态，正文会使用保守表述，例如“可能”“适合用于理解”“需要进一步验证”，而不是写成通用规律。

## 常见误区

- 误区一：Agent 越自治越好。实际系统通常需要在自治、可控性、成本和安全之间取舍。
- 误区二：多 Agent 一定比单 Agent 强。多 Agent 会增加协调成本、调试难度和失败路径。
- 误区三：加了 Memory 就能长期变聪明。错误记忆、过时信息和隐私风险可能让系统更差。
- 误区四：跑通 demo 就代表可靠。Agent 需要 eval、trace、回归测试和权限设计。
- 误区五：Prompt 可以解决所有安全问题。工具权限、数据隔离、人工确认和审计同样重要。

## 本章小结

- 学习 Agent 要先建立概念边界，再读论文和框架。
- References 是正文的一等内容。
- 高可信来源不等于所有结论都可泛化。
- 这本手册会严格区分资料复核、结论验证和工程建议。
- 后续章节会优先面向初学者解释，再逐步进入工程细节。
